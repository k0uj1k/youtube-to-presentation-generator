# Iフレーム抽出高速化 - 最適化レポート

## 実施日
2026-06-14

## 実施内容
YouTube動画のIフレーム抽出処理を高速化するための包括的な最適化を実装しました。

---

## 🚀 主な改善点

### 1. **ビデオスキャン回数の削減（最大効果）**
**問題：**
- 旧実装では動画を**2回フル読み込み**していた
  1. Iフレームの差分計算用に1回
  2. 代表フレーム抽出用に1回（cap = cv2.VideoCapture()で再度開く）

**改善：**
- **ビデオスキャンを完全に1回に統一**
- Iフレームの処理時点でフレームをメモリに保存し、代表フレーム抽出時に再利用
- このため処理時間は**約50%削減**（ビデオI/Oがボトルネックのため）

### 2. **フレームキャッシング機構の導入**
**改善内容：**
```python
# 旧実装: フレームを処理後に破棄
frame, gray = cv2.imread(...)  # 処理
# → 代表フレーム抽出時に再度読み込み
cap.set(cv2.CAP_PROP_POS_FRAMES, frame_idx)
ret, frame = cap.read()  # 重複読み込み

# 新実装: フレームをキャッシュ
frame_storage[ts] = (frame, gray_small)
# → 代表フレーム抽出時にキャッシュから取得（再ロード不要）
```

**効果：**
- ビデオファイルI/Oを大幅削減
- メモリ使用量 vs 速度のバランスを最適化

### 3. **ffprobe処理の改善**
**新しい関数：`get_keyframe_timestamps_cached()`**
- タイムアウトを120秒 → 60秒に短縮
- フォールバック処理を同じVideoCaptureインスタンスで実行
- 不要な再初期化を排除

### 4. **進捗表示とログの充実**
新しい出力：
```
進捗: 50/500 (10%)
進捗: 100/500 (20%)
...
フレーム処理完了: 500 フレーム分析済み
静止区間検出: 15 区間を検出
代表フレームを抽出中...
抽出完了: 12 枚のスライド画像を抽出しました
```

## ⚠️ 注意事項

### メモリ使用量
- Iフレーム画像をメモリに保持するため、メモリ使用量が増加
- 30分動画（150フレーム） × 1920×1080 × 3バイト ≈ 1GBメモリ
- 大規模サーバー環境では気になるかもしれません
- **改善案**: メモリが逼迫した場合はディスクキャッシュに切り替え可能

### 動画ファイルI/O
- SSDでの実行を推奨（HDD環境ではI/Oがボトルネック）
