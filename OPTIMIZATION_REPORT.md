# Iフレーム抽出高速化 - 最適化レポート

## 実施日
2026-06-14

## 実施内容
YouTube動画のIフレーム抽出処理を高速化するための包括的な最適化を実装しました。

---

## 🚀 主な改善点

### 1. **ビデオスキャン回数の削減（最大効果）**
**問題：**
- 旧実装では動画を**2回フル読み込み**していた
  1. Iフレームの差分計算用に1回
  2. 代表フレーム抽出用に1回（cap = cv2.VideoCapture()で再度開く）

**改善：**
- **ビデオスキャンを完全に1回に統一**
- Iフレームの処理時点でフレームをメモリに保存し、代表フレーム抽出時に再利用
- このため処理時間は**約50%削減**（ビデオI/Oがボトルネックのため）

### 2. **基準フレーム比較方式への変更**
**改善内容：**
```python
# 初回フレームを基準フレームとして保存
base_gray_small = gray_small

# 次の I フレームと基準フレームの差分を比較
mad = np.mean(cv2.absdiff(gray_small, base_gray_small))
if mad >= threshold:
    save_scene(frame, ts)
    base_gray_small = gray_small
```

**効果：**
- フル解像度フレームの大量キャッシュを廃止
- 基準フレームとの差分が大きい I フレームを即時スライド化
- メモリ使用量を抑えつつ、スライド切替を直接検出

### 3. **ffprobe処理の改善**
**新しい関数：`get_keyframe_timestamps_cached()`**
- タイムアウトを120秒 → 60秒に短縮
- フォールバック処理を同じVideoCaptureインスタンスで実行
- 不要な再初期化を排除

### 4. **進捗表示とログの充実**
新しい出力：
```
進捗: 50/500 (10%)
進捗: 100/500 (20%)
...
フレーム処理完了: 500 フレーム分析済み
スライド切替検出: 123.45秒 (MAD=12.3)
フレーム処理完了: 500 フレーム処理, 11 回の切替を検出
抽出完了: 12 枚のスライド画像を抽出しました
```

## 📈 今後の改善案（オプション）

### 優先度 高
1. **NumPy最適化**
   - `np.mean()` → `np.mean(diff, dtype=np.float32)` で型を指定
   - ベクトル化操作で計算速度+10%

2. **マルチスレッド処理**
   - Iフレーム読み込みと差分計算を並列化（I/Oと計算の重複）
   - 期待効果：+20～30%

### 優先度 中
3. **GPU加速（オプション）**
   - CUDAでグレースケール変換・リサイズを実行
   - 環境依存性あり

4. **適応的なしきい値調整**
   - I フレーム自体は間引かず、動画ごとの明るさやノイズ量に応じて MAD しきい値を補正
   - 全 I フレームを比較対象にしたまま、過検出と検出漏れのバランスを改善

## ⚠️ 注意事項

### メモリ使用量
- Iフレーム画像をメモリに保持するため、メモリ使用量が増加
- 30分動画（150フレーム） × 1920×1080 × 3バイト ≈ 1GBメモリ
- 大規模サーバー環境では気になるかもしれません
- **改善案**: メモリが逼迫した場合はディスクキャッシュに切り替え可能

### 動画ファイルI/O
- SSDでの実行を推奨（HDD環境ではI/Oがボトルネック）
